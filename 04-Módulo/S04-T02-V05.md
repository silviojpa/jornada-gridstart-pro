## Tarefa 1 — quarto parâmetro STATUS_ESPERADO com default "200"
* O ponto-chave é perceber que essa tarefa é exatamente o mesmo padrão já aplicado a
MAX_TENTATIVAS e INTERVALO no vídeo — só muda o nome da variável, a posição ($4) e o valor
padrão (200). A segunda parte (trocar o "200" fixo dentro do until) é o que de fato muda o
comportamento: sem essa troca, STATUS_ESPERADO existiria mas não teria efeito nenhum.

* Erro comum esperado: declarar STATUS_ESPERADO="${4:-200}" mas esquecer de trocar o "200"
dentro do until — o script "aceita" o argumento mas ignora ele. Vale testar especificamente
o Cenário 2 (status 404) para pegar esse erro: se o aluno não trocou, o script vai tentar
MAX_TENTATIVAS vezes e falhar, mesmo a página respondendo 404 na primeira tentativa.

## Tarefa 2 — mensagem de uso atualizada
<img width="1304" height="851" alt="image" src="https://github.com/user-attachments/assets/0b4e2361-3132-45ab-9223-866647c0fb8b" />
* Tarefa simples, mas importante para reforçar o hábito de manter Uso: sincronizada com os
parâmetros reais do script — uma mensagem de uso desatualizada é pior do que nenhuma, porque
engana quem lê.
Erro comum esperado: colocar STATUS_ESPERADO sem colchetes (<STATUS_ESPERADO>), como
se fosse obrigatório — isso é uma inconsistência leve (não quebra o script), mas vale corrigir no
feedback: ${4:-200} só faz sentido se o argumento for opcional, e a convenção de colchetes existe
para comunicar isso a quem usa o script.

## Tarefa 3 — validação do formato com ${#VARIAVEL}
<img width="1284" height="888" alt="image" src="https://github.com/user-attachments/assets/e059c1c0-dfc6-4f78-80e6-c3ebc369350a" />
* ${#VARIAVEL} retorna o número de caracteres da string armazenada em VARIAVEL. Como todo
código de status HTTP tem 3 dígitos, ${#STATUS_ESPERADO} -ne 3 é uma validação simples e
suficiente para o escopo da atividade — não precisa verificar se são dígitos, só o tamanho.
